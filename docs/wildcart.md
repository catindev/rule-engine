# Работа с массивами и wildcard [*]

## Общая идея

Движок использует **flat‑map модель payload**: все данные представлены как плоский набор ключей `key -> value`.

Пример payload:

```json
{
  "order.id": "A100",
  "checkout.items[0].sku": "APPLE",
  "checkout.items[0].qty": 3,
  "checkout.items[1].sku": "BANANA",
  "checkout.items[1].qty": 2
}
```

В этой модели нет вложенных объектов. Ключ `checkout.items[1].qty` это просто строка. Чтобы писать правила для массивов используется специальный шаблон **wildcard `[*]`**.

# Wildcard [*]

Wildcard используется в поле `field` правила и означает:

> применить правило ко всем элементам массива.

Пример:

```
field: "checkout.items[*].qty"
```

Этот шаблон будет сопоставлен со следующими ключами payload:

```
checkout.items[0].qty
checkout.items[1].qty
checkout.items[2].qty
```

Движок автоматически расширяет wildcard и выполняет правило для каждого совпавшего поля.

# Ограничения wildcard

В текущей версии движка действуют следующие ограничения:

- поддерживается **только один `[*]` в поле**
- wildcard соответствует **только индексам вида `[0]`, `[1]`, `[2]`**
- сопоставление происходит **по точному префиксу и суффиксу**

Например:

```
items[*].qty
```

НЕ совпадет с:

```
checkout.items[0].qty
```

Потому что у ключа другой префикс. Поэтому рекомендуется писать полный путь поля:

```
checkout.items[*].qty
```

**Важно учесть:** Если если риск пропустить проверку массива из-за неверного ключа, то нужно в пайплайне перед правилом проверки массива поставить проверку наличия ключа. Например `checkout.items` поменяли на `items`, а правила проверок поменять забыли и проверки по `checkout.items[*].qty` будут ходить мимо новых полей с массивом просто не находя их. Чтобы этого избежать всегда проектируйте в свих пайплайнах проверку наличия ключа с исключением - так проверки будут выбрасывать ошибки если в них забудут поменять контракт вместо того, чтобы просто игнорировать его изменение.

# Проверка каждого элемента массива (режим EACH)

По умолчанию check‑rules работают в режиме **EACH**.

Это означает:

- правило применяется к каждому элементу
- issue создается для каждого провалившегося элемента

## Пример

```json
{
  "type": "rule",
  "role": "check",
  "field": "checkout.items[*].qty",
  "operator": "greater_than",
  "value": 0,
  "level": "ERROR",
  "code": "ITEM_QTY_INVALID",
  "message": "Количество товара должно быть больше 0"
}
```

Payload:

```json
{
  "checkout.items[0].qty": 3,
  "checkout.items[1].qty": 0
}
```

Результат:

```
issue: checkout.items[1].qty
```

# Агрегация результатов по массиву

Для wildcard‑правил можно указать блок `aggregate`, который определяет как объединять результаты.

```json
"aggregate": { "mode": "ALL" }
```

Поддерживаемые режимы:

| mode  | описание                                         |
| ----- | ------------------------------------------------ |
| EACH  | проверять каждый элемент отдельно (по умолчанию) |
| ALL   | все элементы должны пройти проверку              |
| COUNT | проверка количества элементов прошедших правило  |
| MIN   | агрегировать минимальное значение массива        |
| MAX   | агрегировать максимальное значение массива       |

---

# Режим ALL

Правило должно выполняться для всех элементов массива.

```json
{
  "type": "rule",
  "role": "predicate",
  "field": "checkout.items[*].qty",
  "operator": "greater_than",
  "value": 0,
  "aggregate": {
    "mode": "ALL"
  }
}
```

Семантика:

```
qty[0] > 0 AND qty[1] > 0 AND qty[2] > 0
```

---

# 7. Режим COUNT

COUNT позволяет проверять количество элементов, удовлетворяющих условию.

Пример: "минимум два товара имеют qty > 5".

```json
{
  "type": "rule",
  "role": "predicate",
  "field": "checkout.items[*].qty",
  "operator": "greater_than",
  "value": 5,
  "aggregate": {
    "mode": "COUNT",
    "op": ">=",
    "value": 2
  }
}
```

# Режим MIN

MIN агрегирует минимальное значение массива и проверяет правило для него.

Пример:

"минимальное количество товара должно быть >= 1"

```json
{
  "type": "rule",
  "role": "check",
  "field": "checkout.items[*].qty",
  "operator": "greater_than",
  "value": 0,
  "aggregate": {
    "mode": "MIN"
  }
}
```

Алгоритм:

```
min(items.qty) > 0
```

# Режим MAX

MAX агрегирует максимальное значение.

Пример:

"количество товара не должно превышать 100"

```json
{
  "type": "rule",
  "role": "check",
  "field": "checkout.items[*].qty",
  "operator": "less_than",
  "value": 100,
  "aggregate": {
    "mode": "MAX"
  }
}
```

# Политика для пустого массива

Если wildcard не нашел ни одного поля, используется параметр `onEmpty`.

```json
"aggregate": { "mode": "ALL", "onEmpty": "PASS" }
```

Возможные значения:

| значение  | поведение                      |
| --------- | ------------------------------ |
| PASS      | правило считается пройденным   |
| FAIL      | правило считается проваленным  |
| TRUE      | predicate = TRUE               |
| FALSE     | predicate = FALSE              |
| UNDEFINED | результат неопределен          |
| ERROR     | генерируется ошибка выполнения |

# Рекомендации по проектированию правил

### Всегда используйте полный путь

Хорошо:

```
checkout.items[*].qty
```

Плохо:

```
items[*].qty
```

Иначе правило может не совпасть с payload.

### Используйте EACH для детальных ошибок

Если нужно показать пользователю **какой элемент массива сломан**, используйте режим по умолчанию.

### Используйте MIN/MAX для бизнес‑ограничений

Например:

- "минимальная цена"
- "максимальное количество"

### Используйте COUNT для бизнес‑логики

Примеры:

- "в корзине должно быть минимум 2 товара"
- "не более 5 активных бенефициаров"

# Пример реального payload

```json
{
  "checkout.items[0].sku": "APPLE",
  "checkout.items[0].qty": 3,

  "checkout.items[1].sku": "BANANA",
  "checkout.items[1].qty": 2,

  "checkout.items[2].sku": "ORANGE",
  "checkout.items[2].qty": 1
}
```

Примеры правил:

### Проверка каждого товара

```
checkout.items[*].qty > 0
```

### Все товары должны быть > 0

```
aggregate: ALL
```

### Минимум два товара qty > 1

```
aggregate: COUNT
```

### Минимальное qty > 0

```
aggregate: MIN
```

# Итог

Wildcard `[*]` позволяет безопасно работать с массивами в flat‑map payload без поддержки вложенных JSON структур.

Это дает:

- простую модель данных
- гибкие проверки массивов
- удобство для UI-конструктора правил

# Trace выполнения wildcard

При включенном режиме трассировки (`TRACE=1`) движок добавляет в ответ сводную запись о выполнении wildcard‑правила. Это позволяет проверить, что правило действительно применилось ко всем элементам массива.

Trace включается при запуске сервера:

```bash
TRACE=1 node server.js
```

После этого ответ API будет содержать поле `trace`.

# Сводка wildcard в trace

Для каждого правила с `[*]` движок добавляет запись:

```json
{
  "kind": "TRACE",
  "message": "wildcard aggregate",
  "data": {
    "patternField": "checkout.items[*].qty",
    "aggregateMode": "MAX",
    "matchedCount": 3,
    "matchedSample": [
      "checkout.items[0].qty",
      "checkout.items[1].qty",
      "checkout.items[2].qty"
    ],
    "pickedField": "checkout.items[1].qty",
    "pickedValue": 101,
    "result": "FAIL"
  }
}
```

Описание полей:

| поле          | значение                                                      |
| ------------- | ------------------------------------------------------------- |
| patternField  | исходное поле правила с wildcard                              |
| aggregateMode | режим агрегации (`EACH`, `ANY`, `ALL`, `COUNT`, `MIN`, `MAX`) |
| matchedCount  | сколько полей payload совпало с wildcard                      |
| matchedSample | первые совпавшие ключи payload                                |
| pickedField   | поле выбранное агрегатором (`MIN` / `MAX`)                    |
| pickedValue   | значение выбранного поля                                      |
| result        | итог проверки                                                 |

# Как убедиться что проверены все элементы

Если нужно убедиться, что правило применилось ко всему массиву, необходимо проверить:

```
matchedCount
```

Например:

```
matchedCount = 3
```

означает, что правило было применено к трем элементам:

```
checkout.items[0]
checkout.items[1]
checkout.items[2]
```

Список этих элементов также виден в `matchedSample`.

# Просмотр только wildcard‑проверок

Чтобы увидеть только агрегированные проверки массивов, можно отфильтровать trace:

```bash
curl -s -X POST http://localhost:3000/wildcard_agg_demo \
  -H "Content-Type: application/json" \
  -d @payload.json | jq '.trace[] | select(.message=="wildcard aggregate")'
```

Это покажет только сводные записи wildcard‑агрегаций.

# Типичная проверка трейса при аналитике

Обычно аналитики проверяют:

Какой агрегатор применился?

```
aggregateMode
```

Сколько элементов массива проверено?

```
matchedCount
```

Какие элементы участвовали в проверке?

```
matchedSample
```

Какое значение было выбрано агрегатором (для MIN/MAX)?

```
pickedField
pickedValue
```

Это делает поведение правил полностью прозрачным и исключает спорные ситуации вроде:

> "А вы точно проверили все элементы массива?"

Ответ можно увидеть прямо в trace.
