# Формат данных Flat Payload + Wildcard

## 1. Введение

Данный документ описывает формат входных данных **Flat Payload + Wildcard**, используемый в движке бизнес‑правил.

Цель этого формата — упростить выполнение правил валидации и логики пайплайнов за счёт отказа от вложенных JSON‑структур и использования **плоской модели данных**.

Основные идеи:

- Все данные хранятся в **одном уровне объекта**.
- Поля представлены **строковыми ключами**.
- Точка (`.`) в ключе — **часть имени**, а не оператор навигации.
- Массивы кодируются через **индексы `[0]`, `[1]`, `[2]`**.
- Для обращения ко всем элементам массива используется **wildcard `[*]`**.

Такой подход делает правила проще, ускоряет выполнение и хорошо подходит для интеграционных систем.

---

# 2. Flat Payload

## 2.1 Определение

**Flat payload** — это структура данных вида key‑value, где все значения находятся на верхнем уровне объекта.

Вложенные структуры разворачиваются в строковые ключи.

### Пример

```json
{
  "order.id": "A-100",
  "order.amount": 1200,
  "customer.phone": "+491234567890",
  "meta.createdAt": "2026-03-05T10:00:00Z"
}
```

Здесь ключ

```
order.id
```

— это **буквальное имя поля**, а не путь в объекте.

То есть это **НЕ означает**:

```
payload.order.id
```

В flat payload **нет вложенных объектов**.

---

# 3. Почему используется Flat Payload

## 3.1 Простая логика правил

Правила могут ссылаться на поле напрямую:

```
field: "order.amount"
```

Движку не нужно выполнять навигацию по JSON‑структуре.

---

## 3.2 Удобство интеграций

Flat payload удобен при интеграциях между системами:

- ABS
- CRM
- API Gateway
- BPM / Workflow

Каждая система может преобразовать свои данные в единый flat формат.

---

## 3.3 Простота логирования

Плоские структуры легче логировать и сравнивать.

Пример логов:

```
order.id=A100
order.amount=1200
customer.phone=+491234567890
```

---

# 4. Преобразование вложенного JSON

Исходные данные часто приходят в виде вложенного JSON.

## 4.1 Исходная структура

```json
{
  "order": {
    "id": "A100",
    "amount": 1200
  },
  "customer": {
    "phone": "+491234567890"
  }
}
```

## 4.2 Flat представление

```
order.id = A100
order.amount = 1200
customer.phone = +491234567890
```

Или как JSON:

```json
{
  "order.id": "A100",
  "order.amount": 1200,
  "customer.phone": "+491234567890"
}
```

---

# 5. Представление массивов

Массивы кодируются через **индексную нотацию**.

## 5.1 Пример

Исходный JSON:

```json
{
  "items": [
    { "sku": "APPLE", "qty": 3 },
    { "sku": "BANANA", "qty": 5 }
  ]
}
```

Flat payload:

```
items[0].sku = APPLE
items[0].qty = 3
items[1].sku = BANANA
items[1].qty = 5
```

JSON payload:

```json
{
  "items[0].sku": "APPLE",
  "items[0].qty": 3,
  "items[1].sku": "BANANA",
  "items[1].qty": 5
}
```

---

# 6. Обращение к конкретному элементу массива

Можно проверять конкретный элемент массива.

### Пример правила

```
field: "items[0].sku"
operator: required
```

Это означает:

"у первого элемента массива items должно быть заполнено поле sku".

---

# 7. Wildcard для массивов

Для применения правила ко всем элементам массива используется wildcard `[*]`.

### Синтаксис

```
items[*].qty
```

Это означает:

"применить правило к каждому элементу массива items".

---

# 8. Пример использования Wildcard

Payload:

```json
{
  "items[0].sku": "APPLE",
  "items[0].qty": 3,
  "items[1].sku": "BANANA",
  "items[1].qty": 5
}
```

Правило:

```
field: "items[*].qty"
operator: min
value: 1
```

Интерпретация движком:

```
items[0].qty >= 1
items[1].qty >= 1
```

Правило применяется **к каждому совпавшему ключу**.

---

# 9. Как работает Wildcard

Алгоритм выполнения:

1. Движок получает паттерн

```
items[*].qty
```

2. Паттерн преобразуется в регулярное выражение

```
^items\\[(\\d+)\\]\\.qty$
```

3. Движок ищет совпадения среди ключей payload.

Например:

```
items[0].qty
items[1].qty
items[2].qty
```

4. Правило выполняется для каждого найденного ключа.

---

# 10. Пример из банковского домена

Payload:

```
beneficiaries[0].inn
beneficiaries[0].name
beneficiaries[1].inn
beneficiaries[1].name
```

Правило:

```
field: "beneficiaries[*].inn"
operator: regex
value: "^[0-9]{10}$"
```

Это означает:

"ИНН каждого бенефициара должен состоять из 10 цифр".

---

# 11. Пример сложного payload

```json
{
  "order.id": "A100",
  "order.amount": 1500,

  "customer.id": "C10",
  "customer.phone": "+491234567890",

  "items[0].sku": "APPLE",
  "items[0].qty": 3,

  "items[1].sku": "BANANA",
  "items[1].qty": 5
}
```

Примеры правил:

```
field: "order.amount"
operator: min
value: 1
```

```
field: "customer.phone"
operator: regex
value: "^\\+?[1-9]\\d{7,14}$"
```

```
field: "items[*].qty"
operator: min
value: 1
```

---

# 12. Рекомендации по именованию

Рекомендуется использовать логические префиксы:

```
order.*
customer.*
items[*].*
payment.*
beneficiaries[*].*
```

Это повышает читаемость правил.

---

# 13. Ограничения формата

Flat payload имеет несколько ограничений:

- отсутствуют вложенные структуры
- точки в ключах не являются навигацией
- массивы представлены через индексы

Однако для задач валидации и обработки событий этот формат значительно упрощает архитектуру движка правил.

---

# 14. Итог

Модель **Flat Payload + Wildcard** позволяет:

- упростить DSL правил
- ускорить выполнение
- упростить интеграции
- сделать правила читаемыми

Эта модель особенно хорошо подходит для:

- validation engines
- rule engines
- integration pipelines
- orchestration сервисов

и других систем, где важна декларативная обработка данных.
