# Ближайший план развития движка

## 1. Стабильные идентификаторы проверок (Check IDs)

Каждая проверка в Rule Engine должна иметь стабильный идентификатор (например checkId), который не зависит от конкретного выполнения и не меняется между версиями правил без явного изменения логики проверки. Этот идентификатор используется оркестратором для ветвления логики обработки заявки, а также для хранения результатов проверок как артефактов аудита. Для реализации этого правила переходят от хранения артефактов в файловой системе к хранению правил в базе данных, где каждой проверке присваивается уникальный checkId. Результаты выполнения проверок также могут сохраняться в БД Rule Engine с привязкой к checkId и версии ruleset, что обеспечивает воспроизводимость и возможность анализа выполненных проверок.

## 2. Версионирование набора правил (Ruleset Version)

Каждый запуск проверки должен выполняться в контексте конкретной версии набора правил (rulesetVersion), которая возвращается в ответе Rule Engine. Это позволяет оркестратору сохранять не только результат проверки, но и информацию о том, какой именно набор правил был применён к заявке. Для поддержки этой возможности правила переводятся на хранение в базе данных с механизмом публикации версий ruleset, где каждая публикация формирует иммутабельную версию конфигурации правил. Такая модель обеспечивает воспроизводимость проверок, упрощает аудит и позволяет безопасно обновлять правила без влияния на уже обработанные заявки.

## 3. Явная семантика агрегации для wildcard-путей

Для правил, применяемых к массивам данных (например beneficiary.contacts[*].phone), необходимо определить явную семантику агрегации результатов проверки. Rule Engine должен поддерживать агрегаторы, такие как ANY, ALL, COUNT, MIN, MAX, чтобы однозначно интерпретировать результат проверки для набора элементов массива. Отсутствие явной семантики может привести к неоднозначному поведению правил и ошибкам бизнес-валидации, особенно при расширении модели данных. Поэтому агрегатор должен быть частью определения правила и явно фиксироваться в trace выполнения проверки.

## 4. Разделение payload и контекста выполнения

Формат запроса к Rule Engine должен быть расширен за счёт явного разделения данных, подлежащих проверке (payload), и метаданных выполнения (meta/context). В поле payload передаётся flat-map структура данных заявки, которая непосредственно участвует в проверках. Контекст выполнения (например merchantId, requestType, correlationId) передаётся отдельно в поле meta, аналогично подходу, используемому в архитектуре адаптера к АБС (ЦФТ). Такое разделение предотвращает смешение доменных данных и технических метаданных, упрощает написание правил аналитиками и обеспечивает более прозрачную трассировку выполнения проверок.
