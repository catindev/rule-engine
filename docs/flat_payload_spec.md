# Формат входных данных: JSON и Flat Payload

## 1. Введение

Движок принимает входные данные в двух форматах — **обычный вложенный JSON** и **flat-map**. Оба формата эквивалентны: перед запуском правил движок автоматически конвертирует вложенный JSON в flat-map через `flattenPayload`. Flat-map на входе проходит без изменений (функция идемпотентна).

Аналитик пишет правила используя dot-notation пути — они одинаково работают с обоими форматами входных данных.

---

## 2. Вложенный JSON (рекомендуемый формат)

Клиент отправляет обычный JSON как есть:

```json
{
  "order": {
    "id": "A100",
    "amount": 1200
  },
  "customer": {
    "phone": "+491234567890"
  },
  "items": [
    { "sku": "APPLE", "qty": 3 },
    { "sku": "BANANA", "qty": 5 }
  ]
}
```

Движок сам разворачивает его в flat-map перед прогоном правил:

```
order.id              → "A100"
order.amount          → 1200
customer.phone        → "+491234567890"
items[0].sku          → "APPLE"
items[0].qty          → 3
items[1].sku          → "BANANA"
items[1].qty          → 5
```

---

## 3. Flat-map (альтернативный формат)

Flat-map принимается без изменений — для обратной совместимости и интеграций где данные уже приходят в плоском виде:

```json
{
  "order.id": "A100",
  "order.amount": 1200,
  "customer.phone": "+491234567890",
  "items[0].sku": "APPLE",
  "items[0].qty": 3,
  "items[1].sku": "BANANA",
  "items[1].qty": 5
}
```

---

## 4. Правила конвертации

| Структура | Правило | Пример |
|---|---|---|
| Вложенный объект | ключи соединяются через `.` | `order.id` |
| Массив объектов | индекс в скобках + точка | `items[0].sku` |
| Массив скаляров | индекс в скобках | `countries[0]` |
| Пустой массив | хранится как `[]` | `items: []` |

Ключ `__context` всегда передаётся нетронутым — он зарезервирован для контекста выполнения и недоступен в wildcard-паттернах.

---

## 5. Контекст выполнения

Контекст (`merchantId`, `currentDate` и другие мета-поля) передаётся отдельно в поле `context` запроса — не в `payload`. В правилах контекст доступен через префикс `$context.*`:

```json
{ "field": "$context.merchantId", "operator": "in_dictionary", ... }
```

Такое разделение предотвращает смешение доменных данных и технических метаданных.

---

## 6. Пример запроса

```bash
curl -X POST http://localhost:3000/v1/validate \
  -H 'Content-Type: application/json' \
  -d '{
    "context": {
      "pipelineId": "checkout_main",
      "merchantId": "demo-merchant"
    },
    "payload": {
      "order": { "id": "A100", "amount": 1200, "currency": "RUB" },
      "customer": { "email": "user@example.com" },
      "items": [
        { "sku": "APPLE", "qty": 3 },
        { "sku": "BANANA", "qty": 5 }
      ]
    }
  }'
```

---

## 7. Рекомендации по именованию полей

Используйте логические префиксы соответствующие структуре домена:

```
order.*
customer.*
items[*].*
payment.*
beneficiary.*
accounts[*].*
accounts[*].transactions[*].*
```

Это повышает читаемость правил и упрощает навигацию в UI.
