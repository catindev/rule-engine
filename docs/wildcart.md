# Работа с массивами и wildcard `[*]`

## 1. Общая идея

Движок использует **flat‑map модель payload**: все данные представлены как плоский набор ключей `key -> value`.

Пример payload:

```json
{
  "order.id": "A100",
  "checkout.items[0].sku": "APPLE",
  "checkout.items[0].qty": 3,
  "checkout.items[1].sku": "BANANA",
  "checkout.items[1].qty": 2
}
```

В этой модели **нет вложенных объектов**. Ключ `checkout.items[1].qty` — это просто строка.

Чтобы писать правила для массивов используется специальный шаблон **wildcard `[*]`**.

---

# 2. Wildcard `[*]`

Wildcard используется в поле `field` правила и означает:

> применить правило ко всем элементам массива.

Пример:

```
field: "checkout.items[*].qty"
```

Этот шаблон будет сопоставлен со следующими ключами payload:

```
checkout.items[0].qty
checkout.items[1].qty
checkout.items[2].qty
```

Движок автоматически расширяет wildcard и выполняет правило для каждого совпавшего поля.

---

# 3. Ограничения wildcard

В текущей версии движка действуют следующие ограничения:

- поддерживается **только один `[*]` в поле**
- wildcard соответствует **только индексам вида `[0]`, `[1]`, `[2]`**
- сопоставление происходит **по точному префиксу и суффиксу**

Например:

```
items[*].qty
```

НЕ совпадёт с:

```
checkout.items[0].qty
```

Потому что у ключа другой префикс.

Поэтому рекомендуется писать **полный путь поля**:

```
checkout.items[*].qty
```

---

# 4. Проверка каждого элемента массива (режим EACH)

По умолчанию check‑rules работают в режиме **EACH**.

Это означает:

- правило применяется к каждому элементу
- issue создаётся для каждого провалившегося элемента

## Пример

```json
{
  "type": "rule",
  "role": "check",
  "field": "checkout.items[*].qty",
  "operator": "greater_than",
  "value": 0,
  "level": "ERROR",
  "code": "ITEM_QTY_INVALID",
  "message": "Количество товара должно быть больше 0"
}
```

Payload:

```json
{
  "checkout.items[0].qty": 3,
  "checkout.items[1].qty": 0
}
```

Результат:

```
issue: checkout.items[1].qty
```

---

# 5. Агрегация результатов по массиву

Для wildcard‑правил можно указать блок `aggregate`, который определяет как объединять результаты.

```json
"aggregate": {
  "mode": "ALL"
}
```

Поддерживаемые режимы:

| mode  | описание                                         |
| ----- | ------------------------------------------------ |
| EACH  | проверять каждый элемент отдельно (по умолчанию) |
| ALL   | все элементы должны пройти проверку              |
| COUNT | проверка количества элементов прошедших правило  |
| MIN   | агрегировать минимальное значение массива        |
| MAX   | агрегировать максимальное значение массива       |

---

# 6. Режим ALL

Правило должно выполняться **для всех элементов массива**.

```json
{
  "type": "rule",
  "role": "predicate",
  "field": "checkout.items[*].qty",
  "operator": "greater_than",
  "value": 0,
  "aggregate": {
    "mode": "ALL"
  }
}
```

Семантика:

```
qty[0] > 0 AND qty[1] > 0 AND qty[2] > 0
```

---

# 7. Режим COUNT

COUNT позволяет проверять количество элементов, удовлетворяющих условию.

Пример: "минимум два товара имеют qty > 5".

```json
{
  "type": "rule",
  "role": "predicate",
  "field": "checkout.items[*].qty",
  "operator": "greater_than",
  "value": 5,
  "aggregate": {
    "mode": "COUNT",
    "op": ">=",
    "value": 2
  }
}
```

---

# 8. Режим MIN

MIN агрегирует минимальное значение массива и проверяет правило для него.

Пример:

"минимальное количество товара должно быть >= 1"

```json
{
  "type": "rule",
  "role": "check",
  "field": "checkout.items[*].qty",
  "operator": "greater_than",
  "value": 0,
  "aggregate": {
    "mode": "MIN"
  }
}
```

Алгоритм:

```
min(items.qty) > 0
```

---

# 9. Режим MAX

MAX агрегирует максимальное значение.

Пример:

"количество товара не должно превышать 100"

```json
{
  "type": "rule",
  "role": "check",
  "field": "checkout.items[*].qty",
  "operator": "less_than",
  "value": 100,
  "aggregate": {
    "mode": "MAX"
  }
}
```

---

# 10. Политика для пустого массива

Если wildcard не нашёл ни одного поля, используется параметр `onEmpty`.

```json
"aggregate": {
  "mode": "ALL",
  "onEmpty": "PASS"
}
```

Возможные значения:

| значение  | поведение                      |
| --------- | ------------------------------ |
| PASS      | правило считается пройденным   |
| FAIL      | правило считается проваленным  |
| TRUE      | predicate = TRUE               |
| FALSE     | predicate = FALSE              |
| UNDEFINED | результат неопределён          |
| ERROR     | генерируется ошибка выполнения |

---

# 11. Рекомендации по проектированию правил

### 1. Всегда используйте полный путь

Хорошо:

```
checkout.items[*].qty
```

Плохо:

```
items[*].qty
```

Иначе правило может не совпасть с payload.

---

### 2. Используйте EACH для детальных ошибок

Если нужно показать пользователю **какой элемент массива сломан**, используйте режим по умолчанию.

---

### 3. Используйте MIN/MAX для бизнес‑ограничений

Например:

- "минимальная цена"
- "максимальное количество"

---

### 4. Используйте COUNT для бизнес‑логики

Примеры:

- "в корзине должно быть минимум 2 товара"
- "не более 5 активных бенефициаров"

---

# 12. Пример реального payload

```json
{
  "checkout.items[0].sku": "APPLE",
  "checkout.items[0].qty": 3,

  "checkout.items[1].sku": "BANANA",
  "checkout.items[1].qty": 2,

  "checkout.items[2].sku": "ORANGE",
  "checkout.items[2].qty": 1
}
```

Примеры правил:

### Проверка каждого товара

```
checkout.items[*].qty > 0
```

### Все товары должны быть > 0

```
aggregate: ALL
```

### Минимум два товара qty > 1

```
aggregate: COUNT
```

### Минимальное qty > 0

```
aggregate: MIN
```

---

# Итог

Wildcard `[*]` позволяет безопасно работать с массивами в flat‑map payload без поддержки вложенных JSON структур.

Это даёт:

- простую модель данных
- гибкие проверки массивов
- удобство для будущего UI конструктора правил.
